# PromptBox 信息架构与分阶段实施方案

> 本文基于《[产品设计哲学](./PRODUCT_PHILOSOPHY.md)》制定，用于指导 PromptBox 的结构演进与逐步实现。

## 1. 目标信息架构

PromptBox 应逐步演进为 3 层结构：

### A. 调用层（默认入口）

这是用户最常进入的界面，目标是 **尽快找到并使用 Prompt**。

应包含：

- 搜索框
- 置顶快捷使用区
- 普通结果区
- 简洁详情区
- 立即使用动作

### B. 管理能力（就地完成）

目标是 **稳定维护 Prompt 资产**，但不单独切全局模式。

应包含：

- 新增
- 编辑
- 删除
- 置顶 / 取消置顶
- 分类管理
- 隐藏分类管理

### C. 系统层（设置 / 同步）

这是更低频的系统能力层，目标是 **配置环境，而不是频繁调用**。

应包含：

- 权限说明
- 导入 / 导出
- WebDAV 同步
- 备份 / 恢复
- 快捷键与系统级设置

---

## 2. 当前状态与目标状态的差距

### 当前状态

- 主界面已统一为调用优先：单击 / 回车即用
- 管理动作通过右键、预览按钮、侧栏新增就地完成
- 同步 / 导入导出通过系统菜单进入

### 目标状态

- 列表主路径始终围绕“快速使用”
- 管理动作可用但视觉退后
- 系统能力继续下沉，不抢主流程

---

## 3. 分阶段实施策略

为避免一次性大改导致界面和逻辑失控，采用分阶段实现。

### 第一阶段：调用优先基础（已完成方向）

目标：

- 主界面统一为调用优先
- 管理动作就地可用，不单独切全局模式
- 新增、编辑、删除、置顶不抢主路径

完成标准：

- 单击 / 回车即用
- 右键与预览按钮承担管理
- 系统能力收在设置菜单

### 第二阶段：调用层瘦身

目标：

- 继续压缩默认调用界面的噪音
- 优化搜索、列表、详情区之间的职责边界

本阶段候选动作：

- 让详情区更聚焦“立即使用”
- 让普通卡片更像结果项而不是管理入口
- 优化置顶区与普通区的排序和视觉节奏

### 第三阶段：系统层下沉

目标：

- 把同步、导入导出、权限说明等能力更明确地收纳到系统层

本阶段候选动作：

- 重新组织右上角菜单
- 增加更清晰的系统说明入口
- 让调用页不承担系统级解释职责

### 第四阶段：使用智能化

目标：

- 让 PromptBox 越用越顺手，而不是越用越重

本阶段候选动作：

- 最近使用
- 使用频率排序
- 更优先的置顶策略
- 搜索优先级优化

---

## 4. 当前实现基线

当前产品已经收敛到：

1. 单一主界面，调用优先
2. 单击 / 回车 / ⌘1-9 快速使用
3. 右键与预览按钮管理
4. 预览可收窄 / 折叠，给列表更多空间

---

## 5. 实施约束

为保持产品稳定性，实施过程中遵守以下约束：

1. 不破坏现有快捷键主路径
2. 不降低“点击即用”的效率
3. 优先做结构分层，再做视觉细化
4. 每一步都必须在打包版中验证

---

## 6. 交付方式

后续改造采用以下循环：

1. 文档沉淀
2. 小步实现
3. 打包验证
4. 再进入下一步

这样可以确保 PromptBox 的演进始终沿着清晰的产品哲学前进，而不是在局部体验上不断来回摇摆。

---

## 7. 当前继续推进的实现点

基于前面的分层设计，当前继续往“调用优先”推进两件事：

### A. 执行优先交互

- 卡片是执行入口，不是纯预览入口
- 鼠标单击整张卡片即可使用
- Hover / 焦点负责预览，Click / Enter 负责执行
- Esc 应始终优先服务“退出当前层”
- 列表按置顶 / 最近 / 全部组织，服务肌肉记忆

### B. 使用智能化的第一步

- 记录最近使用时间与使用次数
- 在置顶区之外补一个“最近使用”层
- 让非置顶 Prompt 也能逐渐形成肌肉记忆路径

这一步依然遵守原有约束：

- 不破坏 Alt+E 主路径
- 不牺牲键盘优先
- 不让管理动作重新抢回主界面
