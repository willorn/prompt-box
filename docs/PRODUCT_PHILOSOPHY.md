# PromptBox 产品设计哲学

## 1. 产品一句话定义

PromptBox 不是一个“提示词收藏夹”，而是一个通过全局快捷键被唤起的 **Prompt 快速调用层**。

它的核心价值不是“存了多少提示词”，而是：

> 在用户需要输入的时候，以最低认知成本，把正确的提示词送到当前输入框。

---

## 2. 要解决的核心问题

PromptBox 主要解决的不是“没有 Prompt”，而是以下三个问题：

1. **记不住**
   - 用户知道自己写过好 Prompt，但无法快速回忆原文。

2. **拿不快**
   - Prompt 散落在文档、备忘录、聊天记录、网页收藏夹里，调用链路太长。

3. **进不去**
   - 就算找到了 Prompt，也还要复制、切回应用、粘贴，过程打断当前工作流。

因此，PromptBox 的价值链优先级应当是：

**唤起速度 > 选择速度 > 执行成功率 > 管理能力**

---

## 3. 核心设计原则

### 原则一：调用优先，整理其次

PromptBox 的默认状态必须服务“立即使用”。

- 默认打开即进入调用场景
- 所有高频路径都要围绕“搜索 / 选择 / 使用 / 关闭”
- 整理、编辑、同步、导入导出都属于次级行为

### 原则二：调用优先，管理不抢主路径

主界面始终是调用器：单击 / 回车即用。

- 主路径：搜索 → 选择 → 使用
- 管理动作：右键、预览按钮、顶栏新增
- 系统能力：设置菜单（同步、导入导出、权限）

不需要再切换“使用 / 管理”模式；管理能力应就地可用，但不抢调用主路径。

### 原则三：高频项应该越来越短路

随着使用时间增加，PromptBox 应该越来越像肌肉记忆工具，而不是越来越像资料库。

理想状态：

- 高频 Prompt 进入置顶区
- 置顶区成为一击即用区
- 用户越来越少搜索，越来越少思考“Prompt 放在哪里”

### 原则四：视觉服务于选择成本，而不是装饰

本产品不是展示型产品，视觉的首要目标是：

1. 一眼看清主路径
2. 一眼看清高频区与普通区
3. 一眼看清可执行动作
4. 让非核心动作自动退后

### 原则五：产品的真正主场不在自身窗口，而在外部输入框

PromptBox 自身只是一个短暂停留的中间层。

真正的使用发生在：

- ChatGPT / Claude
- Cursor / VS Code
- 浏览器输入框
- 飞书 / 微信 / 邮件

因此，PromptBox 的成功标准不是“用户在本应用里停留多久”，而是“用户是否更快回到外部输入场景”。

---

## 4. 非目标

PromptBox 当前阶段不应该成为：

- 知识库产品
- 长文档编辑器
- 团队协作平台
- AI 对话客户端
- 复杂工作流编排器

这些方向都会稀释“快速调用层”的产品定位。

---

## 5. 目标用户心智

用户对 PromptBox 的理想感受不是：

- “我在管理 Prompt”

而是：

- “我按一下快捷键”
- “我选一个动作”
- “它立刻帮我把表达能力接上去”

当 PromptBox 做对时，用户几乎不会思考“Prompt 存在哪”，只会思考“我现在要做什么”。

---

## 6. 产品成功标准

从产品哲学出发，PromptBox 的关键指标应围绕调用效率：

1. **调用路径是否足够短**
   - Alt+E → 找到 → 使用 → 回到原应用

2. **执行结果是否稳定**
   - 自动粘贴成功率
   - 失败回退可理解性

3. **高频场景是否越来越顺手**
   - 置顶区使用率
   - 搜索成本是否下降

4. **整理动作是否不干扰使用动作**
   - 主路径是否干净（调用不被管理动作打断）
   - 管理动作是否被合理下沉

---

## 7. 当前版本的设计结论

基于以上哲学，PromptBox 接下来的演进方向应明确为：

> **默认是调用器，管理动作就地可用但不抢主路径，系统设置是更深一层。**

对应到产品结构，就是：

- 调用页（默认）
- 管理动作（右键 / 预览按钮 / 新增，就地完成）
- 设置页 / 系统页（更深层）

这将作为后续所有界面与交互改造的基准。
